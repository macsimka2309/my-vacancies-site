# Что уже сделано и зачем это нужно

Этот документ объясняет текущий каркас проекта простыми словами. Представь проект как здание: сейчас мы сделали фундамент, технические коммуникации и чертеж базы данных. Стены, формы и админку будем добавлять следующими шагами.

## Общая картина

MVP сайта вакансий состоит из трех главных частей:

1. Frontend - страницы, которые видит кандидат и менеджер.
2. Backend - серверная логика, которая принимает отклики, пишет в базу и отправляет Telegram.
3. Database - PostgreSQL, где хранятся вакансии и отклики.

В выбранной архитектуре Next.js закрывает сразу frontend и backend. Это удобно для MVP: один проект, одна сборка, один деплой.

## `package.json`

Файл: `package.json`

Это паспорт Node.js-проекта. В нем указано:

- как проект называется;
- какие библиотеки нужны;
- какие команды можно запускать.

Важные зависимости:

- `next` - фреймворк для сайта и backend endpoint-ов.
- `react` и `react-dom` - библиотека для интерфейса.
- `prisma` - инструмент для описания структуры базы и миграций.
- `@prisma/client` - код, через который приложение будет обращаться к базе.
- `@prisma/adapter-pg` - PostgreSQL-адаптер для Prisma 7.
- `zod` - библиотека для проверки данных из форм.
- `dotenv` - загрузка переменных окружения из `.env`-файлов.

Важные команды:

```bash
pnpm dev
```

Запускает сайт локально в режиме разработки.

```bash
pnpm build
```

Собирает production-версию сайта.

```bash
pnpm start
```

Запускает собранный сайт без режима разработки.

```bash
pnpm typecheck
```

Проверяет TypeScript-типы. Это помогает находить ошибки до запуска.

```bash
pnpm prisma:generate
```

Генерирует Prisma Client. Без этого приложение не сможет удобно обращаться к базе.

```bash
pnpm prisma:migrate
```

Создает и применяет миграции локально во время разработки.

```bash
pnpm prisma:deploy
```

Применяет уже созданные миграции на сервере. Это production-команда.

## `pnpm-lock.yaml`

Файл: `pnpm-lock.yaml`

Это точный список версий всех установленных библиотек. Он нужен, чтобы на твоем компьютере, на сервере и у другого разработчика проект собирался одинаково.

Без lockfile может случиться так: сегодня проект работает, а через месяц подтянулась новая версия зависимости и что-то сломалось.

## `app/`

Папка: `app/`

Это основная папка Next.js App Router. В ней будут страницы сайта и backend route handlers.

Сейчас есть:

```text
app/layout.tsx
app/page.tsx
app/globals.css
app/api/health/route.ts
```

### `app/layout.tsx`

Это общий каркас всех страниц. Все страницы сайта будут вставляться внутрь этого layout.

Там уже задан язык страницы `ru` и подключен общий CSS.

### `app/page.tsx`

Это главная страница сайта. Сейчас там временный экран-заглушка. На следующем шаге сюда логично добавить список активных вакансий.

### `app/globals.css`

Это глобальные стили: фон, шрифты, базовые классы `page-shell`, `panel`, `muted`.

Пока стили минимальные, чтобы не мешать будущей разработке.

### `app/api/health/route.ts`

Это технический endpoint:

```text
GET /api/health
```

Он возвращает простой JSON, например:

```json
{
  "ok": true,
  "service": "salihov-vacancy",
  "timestamp": "2026-07-06T..."
}
```

На сервере это удобно для проверки: приложение живо или нет.

## `next.config.ts`

Файл: `next.config.ts`

Это конфиг Next.js. Сейчас там включено:

```ts
output: "standalone"
```

Это важно для Docker-деплоя. Next.js при сборке положит в `.next/standalone` минимальный набор файлов, который нужен для запуска production-сервера.

Иначе в Docker-образ пришлось бы тащить больше лишних файлов и зависимостей.

## `tsconfig.json`

Файл: `tsconfig.json`

Это настройки TypeScript. TypeScript помогает находить ошибки в коде до запуска приложения.

Например, если функция ждет строку, а мы передали число, TypeScript подсветит проблему.

## `next-env.d.ts`

Файл: `next-env.d.ts`

Это служебный файл Next.js для TypeScript. Его обычно не редактируют руками.

## `lib/db.ts`

Файл: `lib/db.ts`

Это единая точка подключения к базе данных.

Позже, когда нам нужно будет получить вакансии или сохранить отклик, мы будем импортировать `db` из этого файла.

Важная идея: не создавать новое подключение к базе в каждом файле. Лучше иметь один общий helper.

Код также проверяет, что `DATABASE_URL` задан. Если переменная отсутствует, приложение сразу скажет об ошибке.

## `prisma/schema.prisma`

Файл: `prisma/schema.prisma`

Это схема базы данных на языке Prisma. Она описывает, какие таблицы должны быть в PostgreSQL и какие поля в них есть.

Сейчас описаны:

- `Vacancy` - вакансия.
- `Application` - отклик кандидата.
- `LeadStatus` - список возможных статусов лида.

### `Vacancy`

Это таблица `vacancies`.

Она хранит вакансии:

- название;
- проект;
- город;
- формат работы;
- зарплату;
- график;
- обязанности;
- требования;
- условия;
- адрес;
- активна вакансия или нет.

Поле `slug` нужно для красивой ссылки:

```text
/vacancies/kurer-samokat-moskva
```

А не такой:

```text
/vacancies/clx123abc
```

### `Application`

Это таблица `applications`.

Она хранит отклики кандидатов:

- имя;
- телефон;
- возраст;
- комментарий;
- город;
- UTM-метки;
- статус обработки;
- комментарий менеджера;
- была ли отправка в Telegram.

Есть поля `vacancyTitleSnapshot` и `projectSnapshot`. Они нужны, чтобы сохранить название вакансии и проекта на момент отклика. Если потом вакансию переименуют, старый отклик не потеряет исторический контекст.

### `LeadStatus`

Это enum, то есть строгий список допустимых статусов:

- `NEW`
- `IN_PROGRESS`
- `NO_ANSWER`
- `INTERVIEW_DONE`
- `FIT`
- `NOT_FIT`
- `SENT_TO_CLIENT`
- `ACCEPTED`
- `CANDIDATE_REFUSED`
- `DUPLICATE`

Так мы защищаемся от случайных значений вроде `noviy`, `neww`, `новый`, `Новый лид`.

## `prisma/migrations/`

Папка: `prisma/migrations/`

Миграция - это SQL-инструкция, которая создает или изменяет таблицы в базе.

Сейчас добавлена первая миграция:

```text
prisma/migrations/20260706000000_init/migration.sql
```

Она создает:

- PostgreSQL enum `LeadStatus`;
- таблицу `vacancies`;
- таблицу `applications`;
- индексы для ускорения поиска;
- связь отклика с вакансией.

На сервере миграции применяются командой:

```bash
pnpm prisma:deploy
```

или через Docker Compose, как описано в `docs/deployment.md`.

## `prisma/seed-data.ts`

Файл: `prisma/seed-data.ts`

Это тестовые данные для разработки. Сейчас там лежат 3 вакансии:

- курьер в Москве;
- комплектовщик на складе в Казани;
- оператор call-центра на удаленке.

Этот файл не подключается к базе сам. Он просто хранит массив данных.

## `prisma/seed.ts`

Файл: `prisma/seed.ts`

Это скрипт, который берет данные из `prisma/seed-data.ts` и записывает их в PostgreSQL.

Запуск:

```bash
pnpm prisma:seed
```

Скрипт использует `upsert`. Это значит: если вакансия с таким `slug` уже есть, она обновится; если такой вакансии нет, она создастся. Поэтому seed можно запускать повторно без дублей.

Seed-скрипт рассчитан на Node.js 24, потому что запускается напрямую из TypeScript-файла.

## `lib/vacancies.ts`

Файл: `lib/vacancies.ts`

Это слой чтения вакансий. В нем собраны функции для получения вакансий из базы:

- `getActiveVacancies()` - список активных вакансий для главной страницы;
- `getVacancyBySlug(slug)` - одна активная вакансия для страницы вакансии;
- `getActiveVacancySlugs()` - список slug-ов, пригодится для генерации страниц и ссылок.

Такой слой нужен, чтобы страницы сайта не содержали SQL/Prisma-логику напрямую. Страницы будут просто просить: "дай мне активные вакансии", а детали работы с базой будут спрятаны здесь.

## `.gitignore`

Файл: `.gitignore`

Он говорит git, какие файлы не нужно отслеживать.

Например:

- `node_modules/` - установленные зависимости;
- `.next/` - результат сборки;
- `.env` и `.env.production` - секретные настройки;
- `.pnpm-store/` - локальный cache pnpm.

## `.dockerignore`

Файл: `.dockerignore`

Похож на `.gitignore`, но для Docker.

Он говорит Docker, какие файлы не отправлять внутрь build context.

Это ускоряет сборку и не дает случайно положить секреты в Docker-образ.

## `Dockerfile`

Файл: `Dockerfile`

Это инструкция, как собрать production Docker-образ приложения.

В нем несколько стадий:

- `base` - базовая Node.js-среда.
- `deps` - установка зависимостей.
- `builder` - генерация Prisma Client и сборка Next.js.
- `migrator` - отдельная стадия для применения миграций.
- `runner` - минимальный runtime-образ, который запускает сайт.

Зачем несколько стадий? Чтобы в финальный контейнер попало только то, что нужно для запуска, а не весь строительный инструмент.

## `docker-compose.prod.yml`

Файл: `docker-compose.prod.yml`

Это описание production-инфраструктуры из контейнеров:

- `postgres` - база данных PostgreSQL.
- `migrate` - одноразовый контейнер для применения миграций.
- `app` - контейнер с Next.js-приложением.

Приложение слушает порт `3000`, но наружу он проброшен только на `127.0.0.1`. Это значит, что напрямую из интернета порт не открыт. Снаружи к нему должен ходить Nginx.

## `deploy/nginx/salihov-vacancy.conf`

Файл: `deploy/nginx/salihov-vacancy.conf`

Это пример конфига Nginx.

Nginx будет принимать запросы на домен, например:

```text
https://example.com
```

И проксировать их внутрь приложения:

```text
http://127.0.0.1:3000
```

## `public/.gitkeep`

Папка `public/` нужна Next.js для статических файлов: картинок, favicon, robots.txt и так далее.

Файл `.gitkeep` пустой. Он нужен только для того, чтобы git сохранил пустую папку.
